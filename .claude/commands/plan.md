# /plan コマンド

このコマンドは**planner**エージェントを呼び出し、コードを書く前に包括的な実装計画を作成します。

## このコマンドの機能

1. **要件の再確認** - 何を構築する必要があるかを明確にする
2. **リスクの特定** - 潜在的な問題とブロッカーを浮き彫りにする
3. **ステップ計画の作成** - 実装をフェーズに分解する
4. **確認を待つ** - 続行する前にユーザーの明示的な承認を待つ

## 使用するタイミング

`/plan`を使用するのは：
- 新しい機能を開始するとき
- 大きなアーキテクチャ変更を行うとき
- 複雑なリファクタリングに取り組むとき
- 複数のファイル/コンポーネントに影響があるとき
- 要件が不明確または曖昧なとき

## 動作方法

plannerエージェントは：

1. **リクエストを分析**し、要件を明確な言葉で再確認する
2. 具体的で実行可能なステップを持つ**フェーズに分解**する
3. コンポーネント間の**依存関係を特定**する
4. **リスクを評価**し、潜在的なブロッカーを特定する
5. **複雑度を推定**（高/中/低）
6. **計画を提示**し、明示的な確認を待つ

## 使用例

```
User: /plan ユーザーの好みに基づくレコメンド機能を追加したい

Agent (planner):
# 実装計画: 好みベースのレコメンド機能

## 要件の再確認
- ユーザーの好み（辛さ、甘さ、材料）を保存する
- 好みに基づいて料理をフィルタリング・ソートする
- アレルギー情報を考慮する

## 実装フェーズ

### フェーズ1: データモデル
1. **Preferenceモデルの作成** (ファイル: app/models/preference.py)
   - アクション: 好みを表すデータクラスを作成
   - 依存関係: なし
   - リスク: 低

### フェーズ2: ストレージ
2. **LocalStorage連携** (ファイル: app/static/js/preferences.js)
   - アクション: 好みの保存・読み込み機能
   - 依存関係: フェーズ1
   - リスク: 低

### フェーズ3: フィルタリングロジック
3. **レコメンドサービス** (ファイル: app/services/recommendation.py)
   - アクション: 好みに基づくスコアリング・フィルタリング
   - 依存関係: フェーズ1
   - リスク: 中

### フェーズ4: UI統合
4. **好み設定UI** (ファイル: app/templates/components/preferences.html)
   - アクション: 好み入力フォーム
   - 依存関係: フェーズ2
   - リスク: 低

## リスク
- 中: スコアリングアルゴリズムの調整が必要かもしれない
- 低: LocalStorageの容量制限

## 成功基準
- [ ] 好みを保存・読み込みできる
- [ ] 料理が好みに基づいてソートされる
- [ ] アレルギー警告が表示される

**確認待ち**: この計画で進めますか？(yes/no/modify)
```

## 重要な注意事項

**重要**: plannerエージェントは、ユーザーが明示的に「yes」「proceed」「進めて」などの肯定的な応答で計画を確認するまで、**コードを一切書きません**。

変更が必要な場合は以下のように応答：
- "modify: [変更内容]"
- "別のアプローチ: [代替案]"
- "フェーズ2をスキップしてフェーズ3を先に"

## 他のコマンドとの統合

計画後：
- `/tdd`を使用してテスト駆動開発で実装
- `/build-fix`を使用してビルドエラーが発生した場合に修正
- `/code-review`を使用して完成した実装をレビュー
