# Issue 125 ロードマップからの作業開始

Issue 125 (API Issue 依存関係マップ) に記載されている実装推奨順序に基づき、次に着手すべきIssueを特定して作業を開始します。

## 手順

1.  **ロードマップの確認**
    *   `gh issue view 125` コマンドを実行し、Issue 125の本文（依存関係図と推奨順序テーブル）を取得してください。

2.  **未完了Issueの特定**
    *   テーブルに記載されている各Issue（例: #107, #108...）について、現在のステータスを確認してください。
    *   `gh issue view {Issue番号} --json state` コマンドを使用して、ステータスが `OPEN` であるものを抽出してください。

3.  **優先順位の決定**
    *   未完了（OPEN）のIssueの中で、以下の基準に従って最も優先すべきIssueを1つ選定してください。
        1.  フェーズが早いもの（フェーズ1 > フェーズ2 > ...）
        2.  優先度が高いもの（High > Medium > Low）
        3.  テーブルの上から順

4.  **Issueの内容確認**
    *   選定したIssueの内容（タイトル、本文）を表示し、ユーザーに「このIssueに着手しますか？」と確認してください。

5.  **ブランチの作成**
    *   ユーザーの承認が得られた場合、新しいブランチを作成してチェックアウトしてください。
    *   **命名規則**: `feature/issue-{Issue番号}-{簡易タイトル}`
        *   例: Issue #107 "レスポンス形式の統一" -> `feature/issue-107-unify-response-format`
        *   簡易タイトルは英語のケバブケースに変換してください。

6.  **Issue 125の更新（作業開始）**
    *   `gh issue comment 125 --body "Issue #{選定したIssue番号} の実装を開始しました。"` を実行し、ロードマップIssueに作業開始を記録してください。

7.  **実装の開始（start-implemention.md 準拠）**
    *   ブランチ作成後、以下のルールに従って実装を進めてください。

    **実装の流れ:**
    1.  `gh issue view {Issue番号}` でIssueの詳細を再確認する。
    2.  実装内容を分析し、**具体的な手順をTODOリストとして書き出す**。
    3.  TODOリストに沿って、関連する既存のソースコードを参考にしながら実装を進める（完了したタスクには都度チェックを入れる）。
    4.  意味のある単位で段階的にコミットする。
        *   メッセージ例: "feat: XXX機能を追加", "fix: XXXのバグ修正"

    **品質基準:**
    *   コードは既存のスタイルとコーディング規約に合わせる。
    *   適切なエラーハンドリングを実装する。
    *   必要に応じてコメントやドキュメントを追加する。
    *   関連するテストがあれば更新/追加する。

8.  **開始報告**
    *   「Issue #{番号} の作業を開始しました。ブランチ: {ブランチ名}。TODOリストを作成して実装に入ります。」と報告してください。

※ 実装完了後は `merge-issue` コマンドを使用してPR作成とマージを行ってください。
