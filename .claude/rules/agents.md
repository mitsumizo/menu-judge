# エージェント活用ガイド

## 利用可能なエージェント

`.claude/agents/`に配置：

| エージェント | 目的 | 使用タイミング |
|-------------|------|---------------|
| planner | 実装計画 | 複雑な機能、リファクタリング |
| tdd-guide | テスト駆動開発 | 新機能、バグ修正 |
| code-reviewer | コードレビュー | コード作成後 |
| security-reviewer | セキュリティ分析 | コミット前 |

## 即座にエージェントを使用

ユーザーのプロンプトを待たずに：

1. **複雑な機能リクエスト** → **planner**エージェントを使用
2. **コードを書いた/変更した直後** → **code-reviewer**エージェントを使用
3. **バグ修正や新機能** → **tdd-guide**エージェントを使用
4. **セキュリティに関わる変更** → **security-reviewer**エージェントを使用

## 並列タスク実行

独立した操作には常に並列Task実行を使用：

```markdown
# 良い例: 並列実行
3つのエージェントを並列で起動：
1. エージェント1: claude_provider.pyのセキュリティ分析
2. エージェント2: menu.pyのコードレビュー
3. エージェント3: test_services.pyの型チェック

# 悪い例: 不必要なシーケンシャル実行
まずエージェント1、次にエージェント2、そしてエージェント3
```

## ワークフローパターン

### 新機能開発

```
1. /plan → 実装計画を作成
   ↓
2. /tdd → テスト駆動で実装
   ↓
3. /code-review → 品質チェック
   ↓
4. /security-check → セキュリティ検証
   ↓
5. コミット＆プッシュ
```

### バグ修正

```
1. /tdd → バグを再現するテストを書く
   ↓
2. 最小限の修正を実装
   ↓
3. /code-review → 修正をレビュー
   ↓
4. コミット＆プッシュ
```

### リファクタリング

```
1. /plan → リファクタリング計画
   ↓
2. 既存のテストが通ることを確認
   ↓
3. 段階的にリファクタリング
   ↓
4. /code-review → 変更をレビュー
   ↓
5. コミット＆プッシュ
```

## マルチパースペクティブ分析

複雑な問題に対しては、複数の視点からサブエージェントを使用：

- **事実確認者**: コードが仕様通りか
- **シニアエンジニア**: アーキテクチャとパターン
- **セキュリティエキスパート**: 脆弱性
- **一貫性確認者**: スタイルと規約
- **冗長性チェッカー**: 重複コード

## エージェント選択の判断基準

| シナリオ | 推奨エージェント |
|---------|-----------------|
| 「〜を実装したい」 | planner → tdd-guide |
| 「バグがある」 | tdd-guide |
| 「コードを書いた」 | code-reviewer |
| 「セキュリティが心配」 | security-reviewer |
| 「ビルドが失敗」 | /build-fix |
| 「リファクタリングしたい」 | planner → code-reviewer |
